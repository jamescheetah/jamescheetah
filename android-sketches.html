Review Checkout

The checkout view can be accessed by tapping on the "Checkout" button on the Cart view. Several elements are modifiable: The delivery date, order note, delivery note, and promo code state. Elements that are not modifiable: Payment method and Delivery Address.

Added Promo Code Option 1: If a customer adds a promo code that reduces the cost of the order, we can show how much they are saving as a separate line item. Note: I changed the link text to "Change Promo Code" to emphasize to the user that their promo code was added but perhaps we could use an alert instead.

Added Promo Code Option 2: If the promo codes we're using just reduce the cost of shipping, we can instead just change the cost of shipping to $0.00 while continuing the practice of altering the promo code link text.

For same-day deliveries, we could just show one line that has the full delivery cost.

Credit Card Option 1: If the credit card processing fee is applied to the entire sum that includes the delivery fee, then the credit card processing fee can be separated by lines and shown right before the final total.

Credit Card Option 2: If the credit card processing fee is only applied to the subtotal, then it can be shown right after the subtotal and right before the delivery fee.

Monthly Subscription: For monthly subscriptions, the fee can be shown in the table above the delivery fee, which would be set to $0.00 (to emphasize that their subscription is paying for free deliveries). We can also show the month if it's helpful to emphasize that it only happens once a month.

Checkout Utilities - Delivery Dates

We can show the two fastest deliveries at the top (with slightly bigger font size), and have the list of remaining days below. Days which have 0 spots left are grayed out.

Reserved Window - 1: If there's really only one reserved window per week, we can show it at the very top as it's own selection. And instead of showing how many spots left below, we show the day of the week.

Reserved Window - 2: Alternatively, we can replace specific days/windows with reserved windows (assuming reserved windows are both specific days and hours), with a slightly different visual treatment that replaces the spots left. This makes sense especially if there's no reason to show duplicate days of the week with varying delivery windows.

Import Order History - Restaurant Depot Account

If there are 0 items in the order guide (or few items) then a message will appear prompting the user to import their order history to auto-populate their order guide.

After tapping the Create My Guide in the previous screen, a window will appear with two choices: the ability to import their Restaurant Depot order history or their order histroy from a non-Restaurant Depot provider.

If the user elects to import their RD order history, they'll be shown a screen where they can enter their RD account number. This is what it looks like with no parameters entered. Also note that they keyboard should be the numeric version.

What it looks like when the user enters a sample account number.

If the user is having trouble entering their account number, they can tap on Having Trouble (previous screen) which will show a pop-up asking them if they'd like for one of our representatives to reach out to them. Tapping yes should send an automated message to one of our customer service reps.

If they tapped yes on the previous screen, a confirmation window will appear.

Assuming that the account number was entered correctly and we were able to retrieve their order history, a counter will appear showing how many items were found. Tapping Finish should bring them back to their newly created Order Guide.

Import Order History - Non-Restaurant Depot Account

In this scenario, we'll assume the user wants to import their order guide but wants to import a non-Restaurant Depot order history. In that case, they would select the second option.

From this view, they are instructed to take a photo of their invoices and send them to us. Tapping either the camera or the library should open the Android OS' version of the camera/library, and selecting a photo should automatically replace the sample image and show a loading animation, automatically uploading the photograph to our server.

After a single image has been successfully sent to us, the user can tap on Send Another Invoice to go back to the previous screen to continue sending more invoices, or tap on Close to go back to the part of the app they were before.

My Staff

My Staff is a section where the owner can invite people to join the app and where they can manage people on the team. When there are 0 members, a temporary message appears, with the options to invite people either by text or e-mail.

If inviting by text, the To field is automatically highlighted and a pre-defined text message is shown. Keep in mind that this design doesn't have to look exactly this way but can re-use the existing model of how text messaging would get prompted in an Android app. The sketch is merely used to denote the functions that should appear on the screen. Additional functions that would be helpful is maybe the ability to open the address book. As the user types a phone number or name of a person in their contact list, a list of auto-suggested contacts should appear, not unlike how it works for text messaging apps.

In the case of e-mail, there might be more fields (e.g. subject, which mailbox to use), but the idea should be the same: re-using system elements alongside a pre-written message to allow for easy creation of an e-mail that will get sent out to someone in the users e-mail contact list.

If is an invited user has installed the app, they will appear on the list of My Staff. Invited users who haven't installed the app yet will also appear on the list, except with explanatory text describing the fact that they aren't part of the team just yet. The user can either tap on the X to remove the user from the team, or tap the circular icon to resend an invite.
